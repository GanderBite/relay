# Race Package Format

Every race — whether installed from the catalog, generated by `relay new`, or hand-written — has the same directory shape. This document is the authoritative reference for that shape. All sample races (`hello-world`, `hello-world-mocked`, `codebase-discovery`) and every user-authored race must conform to it.

---

## Directory layout

```
<race-name>/
├── package.json           # name, version, peer-dep on @relay/core
├── race.ts                # defineRace() entry point — default export
├── prompts/
│   ├── 01_<runner>.md     # prompt file for each runner.prompt
│   ├── 02_<runner>.md
│   └── ...
├── schemas/               # optional: shared Zod schemas
│   └── <name>.ts
├── templates/             # optional: output templates (HTML, markdown)
│   └── <name>.<ext>.ejs
├── examples/              # optional: sample outputs for the README
│   └── sample-output.<ext>
├── README.md              # user-facing docs (see "README sections" below)
└── tsconfig.json          # extends @relay/core/tsconfig
```

The `prompts/`, `schemas/`, `templates/`, and `examples/` directories are optional. `package.json`, `race.ts`, and `README.md` are required. A race without all three will not pass `relay test` or catalog lint.

---

## package.json

Required fields and the `relay` metadata block:

```json
{
  "name": "@ganderbite/race-codebase-discovery",
  "version": "0.1.0",
  "description": "Explores a codebase and produces an HTML report.",
  "type": "module",
  "main": "./dist/race.js",
  "files": ["dist", "prompts", "schemas", "templates", "examples", "README.md"],
  "scripts": {
    "build": "tsc",
    "test": "relay test ."
  },
  "peerDependencies": {
    "@relay/core": "^1.0.0"
  },
  "relay": {
    "displayName": "Codebase Discovery",
    "tags": ["discovery", "documentation"],
    "estimatedCostUsd": { "min": 0.20, "max": 0.80 },
    "estimatedDurationMin": { "min": 5, "max": 20 },
    "audience": ["pm", "dev"]
  }
}
```

**Package naming.** Catalog race packages are named `@ganderbite/race-<name>` — the `race-` prefix is mandatory. The CLI's `relay install` command resolves a bare `<name>` to `@ganderbite/race-<name>`. A race named `codebase-discovery` is published as `@ganderbite/race-codebase-discovery`.

**The `relay` metadata block** is read by the CLI's `list`, `search`, and pre-run banner. It is not consumed by `@relay/core` itself — the runner loads `race.ts`, not `package.json`. All five keys (`displayName`, `tags`, `estimatedCostUsd`, `estimatedDurationMin`, `audience`) are required. Catalog lint reports an error if any are missing.

## race.ts entry point

`race.ts` must default-export a `Race` object returned by `defineRace`. Both `defineRace` and the `runner.*` builders throw `RaceDefinitionError` synchronously when their configuration is invalid — validation happens at import time, not at runtime, so race.ts can export the result directly with no `Result` unwrapping.

```ts
import { defineRace, runner, z } from '@relay/core';

export default defineRace({
  name: 'codebase-discovery',
  version: '0.1.0',
  description: 'Explores a codebase and produces an HTML report.',
  input: z.object({
    repoPath: z.string(),
    audience: z.enum(['pm', 'dev', 'both']).default('both'),
  }),
  runners: {
    inventory: runner.prompt({
      promptFile: 'prompts/01_inventory.md',
      output: { baton: 'inventory' },
    }),
    entities: runner.prompt({
      promptFile: 'prompts/02_entities.md',
      dependsOn: ['inventory'],
      contextFrom: ['inventory'],
      output: { baton: 'entities' },
    }),
    report: runner.prompt({
      promptFile: 'prompts/03_report.md',
      dependsOn: ['entities'],
      contextFrom: ['inventory', 'entities'],
      output: { artifact: 'report.html' },
    }),
  },
});
```

Rules:

- The `name` field should match the package's bare name (`codebase-discovery`, not the full scoped name).
- The `version` in `defineRace` is the race's own version — it is independent from `package.json`'s version, but in practice they track together.
- The CLI loads the compiled `dist/race.js` via dynamic ESM `import()`. Races are compiled to JS at publish time. The CLI never runs `tsc` on install.
- `runner.prompt`, `runner.script`, `runner.branch`, `runner.parallel`, and `runner.terminal` are the five runner constructors. Each validates its config at build time and throws `RaceDefinitionError` on bad input — a malformed runner aborts module loading before any run can start.

## README sections

Every race's README must contain the following sections, in this order:

1. **What it does** — one paragraph describing the race's purpose and output.
2. **Sample output** — an image or excerpt showing what the race produces.
3. **Estimated cost and duration** — numbers, not adjectives. Match the `relay` metadata block in `package.json`.
4. **Install command** — `relay install <name>`.
5. **Run command** — `relay run <name> <input>` with the most common arguments shown.
6. **Configuration** — what inputs and options the race exposes.
7. **Environment** (optional) — auth expectations and env requirements specific to this race. Races that run on the Claude subscription should point at `docs/billing-safety.md`; mocked races should note that no API key or subscription is required.
8. **Customization guide** — how to fork and adapt the race.
9. **License**.

Sections 1–5 are mandatory. The catalog linter reports an error for any missing mandatory section and rejects the race from the catalog homepage. Sections 6–9 are optional but strongly encouraged; the linter reports a warning if the mandatory subset is absent. The optional "Environment" section, when present, must appear between Configuration and Customization — the linter flags it in any other position.

Example heading block:

```markdown
## What it does

## Sample output

## Estimated cost and duration

## Install

## Run

## Configuration

## Environment

## Customization

## License
```

The linter matches on these heading strings, so keep them exact. Section 5 ("Run") is the heading most often renamed — "Usage" will not pass lint.

## Peer dependency on @relay/core

Race packages declare `@relay/core` as a `peerDependency`, not a direct dependency:

```json
"peerDependencies": {
  "@relay/core": "^1.0.0"
}
```

This means the user's project (or the CLI host) provides the `@relay/core` instance — the race does not bundle its own copy. The caret range (`^1.0.0`) pins to a compatible major. Do not use `*` or an open range; the catalog lint treats an unpinned peer as a warning.

When `@relay/core` reaches `1.0.0`, races published against `^1.0.0` continue to work across minor and patch releases. A breaking change to `@relay/core`'s public API requires a new major and a coordinated catalog update. Until `@relay/core` is at `1.0.0`, use an exact or tilde range (`~0.x.y`) to avoid surprises on patch bumps during the pre-release period.

## Env containment — prompt vs. script/branch runners

`runner.prompt` runs in a contained subprocess. The runner builds an explicit env allowlist (`PATH`, `HOME`, `USER`, `LANG`, `TZ`, `TMPDIR`, `SHELL`, `CLAUDE_*`, opt-in `ANTHROPIC_*`) and passes only those variables to the Claude Agent SDK. `ANTHROPIC_API_KEY` is suppressed from the prompt subprocess unless the caller explicitly opted in. This is the §8.1 safety contract — see `docs/billing-safety.md` for the full guard protocol.

`runner.script` and `runner.branch` forward `process.env` intact to the spawned shell. Every variable on the machine — including `ANTHROPIC_API_KEY`, `AWS_SECRET_ACCESS_KEY`, `GITHUB_TOKEN`, and any other credential — is visible to the child process.

⚠ This is by design. Script and branch runners are user-controlled shell. If your script touches the network or an external tool, unset sensitive variables in a wrapper or pass an explicit `env` map:

```ts
runner.script({
  run: 'curl -s -X POST $WEBHOOK_URL -d @batons/summary.json',
  env: { WEBHOOK_URL: process.env.WEBHOOK_URL ?? '' },
  // ANTHROPIC_API_KEY and other credentials are still visible
  // because runner.script merges env on top of process.env.
})
```

`relay doctor` surfaces whether `ANTHROPIC_API_KEY` is set before any race runs, giving you a chance to unset it or opt in explicitly before the runner reaches a script runner.

See `docs/billing-safety.md` for the full env passthrough rules, the `RELAY_ALLOW_API_KEY` opt-in mechanism, and CI configuration guidance.

## Versioning

Every race follows strict semver:

| Change | Bump |
|---|---|
| Breaking change to the `input` schema | Major |
| Removal of an artifact or baton | Major |
| Rename of a public output (baton or artifact key) | Major |
| Adding optional inputs | Minor |
| Adding new artifacts | Minor |
| Prompt edits, bug fixes | Patch |

The catalog client respects npm semver. `relay install codebase-discovery@^0.1.0` installs any compatible version. A race author who makes a breaking input change must bump the major and publish both versions until consumers migrate.
